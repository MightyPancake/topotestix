# Research: PostgreSQL streaming-replication SUT for topotestix

## Summary
NixOS does **not** expose a dedicated `services.postgresql.replication` subtree; streaming-replication tuning is done through the generic `services.postgresql.settings` attrset plus `authentication`/`enableTCPIP`, while backups live in a separate `services.postgresqlBackup` module. PostgreSQL’s physical-replication surface is dominated by a small set of interacting GUCs (`wal_level`, sender/slot limits, sync-commit settings, WAL-retention knobs, hot-standby conflict knobs, and `primary_conninfo`/`primary_slot_name`), which gives a good fuzzing surface for liveness, lag, and failover properties. 

## Findings
1. **NixOS surface: no replication-specific subtree** — In current nixpkgs `master`, `services.postgresql` exposes `enable`, `package`, `checkConfig`, `dataDir`, `authentication`, `identMap`, `initdbArgs`, `initialScript`, `ensureDatabases`, `ensureUsers`, `enableTCPIP`, `extensions`, `settings`, and `superUser`; `services.postgresql.settings` is a freeform `postgresql.conf` surface with documented suboptions `shared_preload_libraries`, `log_line_prefix`, and `port`, while the implementation forces `hba_file`, `ident_file`, `listen_addresses`, and `jit`. There is no `services.postgresql.replication` option in `master` or in `release-24.11`/`release-24.05`—replication is configured by writing raw GUCs into `settings`. [master file](https://github.com/NixOS/nixpkgs/blob/master/nixos/modules/services/databases/postgresql.nix#L197-L639), [release-24.11 file](https://github.com/NixOS/nixpkgs/blob/release-24.11/nixos/modules/services/databases/postgresql.nix#L360-L500), [release-24.05 file](https://github.com/NixOS/nixpkgs/blob/release-24.05/nixos/modules/services/databases/postgresql.nix#L360-L500)
2. **NixOS version drift matters** — `release-24.11` and `release-24.05` still expose the legacy `extraPlugins` name and a `recoveryConfig` option; `master` has renamed plugins to `extensions` and removes `recoveryConfig` (PostgreSQL 12+ style `standby.signal`/`postgresql.conf` recovery). `services.postgresqlBackup` is a separate module (`nixos/modules/services/backup/postgresql-backup.nix`) with `enable`, `startAt`, `backupAll`, `databases`, `location`, `pgdumpOptions`, `pgdumpAllOptions`, `compression`, and `compressionLevel`—it is pg_dump/pg_dumpall based, not physical replication-aware. [master file](https://github.com/NixOS/nixpkgs/blob/master/nixos/modules/services/databases/postgresql.nix#L430-L560), [postgresql-backup.nix](https://github.com/NixOS/nixpkgs/blob/master/nixos/modules/services/backup/postgresql-backup.nix#L76-L133)
3. **`wal_level` / sender limits / slots are the core physical-replication knobs** — `wal_level=replica` is enough for physical standby + archiving; `logical` adds logical decoding, and `minimal` is incompatible with PITR/standbys (and the server will not start with nonzero `max_wal_senders`). `max_wal_senders` caps concurrent standby/basebackup clients and `0` disables replication; `max_replication_slots` caps slot count and must be high enough for the cluster’s existing slots. [wal_level](https://www.postgresql.org/docs/18/runtime-config-wal.html), [max_wal_senders / max_replication_slots](https://www.postgresql.org/docs/current/runtime-config-replication.html), [warm standby setup](https://www.postgresql.org/docs/18/warm-standby.html)
4. **WAL-retention knobs interact tightly** — `wal_keep_size` prevents recycling WAL too early for a slow standby, but if the standby falls behind more than that size the connection can be terminated and downstreams fail too; the doc recommends a replication slot or archive instead. `archive_mode` + `archive_command` provide an alternate catch-up path, but only if archiving succeeds. `max_wal_size`/`min_wal_size` affect how aggressively `pg_wal` segments are recycled. [replication settings](https://www.postgresql.org/docs/current/runtime-config-replication.html), [continuous archiving](https://www.postgresql.org/docs/18/continuous-archiving.html), [WAL configuration](https://www.postgresql.org/docs/18/wal-configuration.html)
5. **Synchronous replication is a commit-latency / availability trade-off** — `synchronous_standby_names` enables synchronous replication, and `synchronous_commit` controls whether commits wait for local flush, remote write, or remote apply. `remote_apply` waits for standby replay (visible reads) and can block all commits if the synchronous standby is down or never catches up; `remote_write` is weaker but still raises latency. Priority/quorum syntax (`FIRST` / `ANY`) changes which standby(s) gate commits. [runtime-config-replication](https://www.postgresql.org/docs/18/runtime-config-replication.html), [warm standby synchronous replication](https://www.postgresql.org/docs/18/warm-standby.html)
6. **Standby read-only behavior and conflict knobs are a rich negative-property source** — `hot_standby=on` allows read-only queries during recovery; standby writes, DDL, `VACUUM`, `CREATE INDEX`, `DROP DATABASE`, `SELECT ... FOR UPDATE`, etc. are rejected. `max_standby_streaming_delay` / `max_standby_archive_delay` bound how long replay may wait before canceling conflicting standby queries, and `hot_standby_feedback` suppresses some cancels at the cost of primary bloat. [hot standby](https://www.postgresql.org/docs/18/hot-standby.html), [replication settings](https://www.postgresql.org/docs/current/runtime-config-replication.html)
7. **Failover/promotion is explicit and timeline-sensitive** — A standby enters recovery when `standby.signal` exists; `pg_ctl promote` or `pg_promote()` exits standby mode. For multi-standby or failover-friendly setups, `recovery_target_timeline='latest'` matters so downstreams follow the new timeline after promotion. PostgreSQL 12 removed `recovery.conf`; keeping old file-based recovery configs is a hard startup failure. [warm standby](https://www.postgresql.org/docs/18/warm-standby.html), [pg_ctl](https://www.postgresql.org/docs/18/app-pg-ctl.html), [recovery.conf removed](https://www.postgresql.org/docs/18/recovery-config.html)
8. **Recent releases add failover and observability features that matter** — PG16 added logical decoding on standbys / logical replication from standbys; PG17 added logical replication failover control, `pg_createsubscriber`, and `pg_upgrade` preservation of logical slots; PG18 adds `sync_replication_slots`, `pg_sync_replication_slots`, logical-slot failover controls, write-conflict reporting for logical replication, and `CREATE SUBSCRIPTION` defaults to parallel streaming. PG18.1 fixed incorrect replication-lag reporting in `pg_stat_replication`, and PG18.2 fixed a race that could invalidate newly created/synced replication slots. [release 16](https://www.postgresql.org/docs/16/release-16.html), [release 17](https://www.postgresql.org/docs/17/release-17.html), [release 18](https://www.postgresql.org/docs/18/release-18.html), [18.1 fix](https://www.postgresql.org/docs/18/release-18-1.html), [18.2 fix](https://www.postgresql.org/docs/release/18.2/)

## Findings (config interaction / failure modes)
1. **Replica lag / eventual consistency** — Asynchronous streaming replication is normal; the standby can lag behind by under a second in the happy case, but heavy WAL production or network/CPU pressure increases lag. [warm standby monitoring](https://www.postgresql.org/docs/18/warm-standby.html)
2. **WAL removed before standby catches up** — If `wal_keep_size` is too small or no slot/archive protects the WAL, the standby can lose its WAL horizon and require a fresh base backup. [warm standby](https://www.postgresql.org/docs/18/warm-standby.html), [replication settings](https://www.postgresql.org/docs/current/runtime-config-replication.html)
3. **Primary commit stalls under sync replication** — With synchronous replication, especially `remote_apply`, a down/slow standby can block or severely delay commits; `synchronous_standby_names` governs who counts. [runtime-config-replication](https://www.postgresql.org/docs/18/runtime-config-replication.html)
4. **Standby query conflicts / cancellations** — Vacuum cleanup, page visibility-map updates, DDL, table locks, and database/table drops can cancel standby queries unless replay is delayed. [hot standby conflicts](https://www.postgresql.org/docs/18/hot-standby.html)
5. **Primary bloat from `hot_standby_feedback`** — Feedback avoids some standby cancels, but delays dead-row cleanup on the primary and can produce bloat, especially across cascaded replicas. [runtime-config-replication](https://www.postgresql.org/docs/current/runtime-config-replication.html), [hot standby conflicts](https://www.postgresql.org/docs/18/hot-standby.html)
6. **Cascading replication lag propagation** — Downstream standbys depend on the upstream having WAL; if the upstream disconnects or falls behind, lag propagates and downstreams may fail to continue. [warm standby cascading replication](https://www.postgresql.org/docs/18/warm-standby.html)
7. **Promotion/timeline divergence** — After failover/promote, downstreams must follow the new timeline; wrong `recovery_target_timeline` or stale config can strand replicas on the old branch. [warm standby](https://www.postgresql.org/docs/18/warm-standby.html)
8. **Recovery-config migration failures** — Leaving `recovery.conf` around on PG12+ is a hard startup failure; `standby.signal` is the supported mechanism. [recovery.conf removed](https://www.postgresql.org/docs/18/recovery-config.html)

## Findings (realistic property assertions for psql-based integration tests)
1. **Primary is writable, standby is read-only** — `INSERT/UPDATE/DELETE` succeeds on primary and fails on standby; `SHOW transaction_read_only` (or `SHOW in_hot_standby`) is `off` on primary and `on` on standby. [hot standby](https://www.postgresql.org/docs/18/hot-standby.html)
2. **Replication eventually converges** — After K writes on primary, the same row count / checksum appears on standby within N seconds. [warm standby monitoring](https://www.postgresql.org/docs/18/warm-standby.html)
3. **Lag is bounded / measurable** — Compare `pg_current_wal_lsn()` on primary to `pg_last_wal_receive_lsn()` or `pg_last_wal_replay_lsn()` on standby; assert the byte gap or time lag stays below a threshold. [warm standby monitoring](https://www.postgresql.org/docs/18/warm-standby.html)
4. **Exactly one node is read/write at a time** — In steady state, only the primary should accept writes; after promotion, only the promoted node should accept writes. [hot standby](https://www.postgresql.org/docs/18/hot-standby.html), [pg_ctl promote](https://www.postgresql.org/docs/18/app-pg-ctl.html)
5. **Promotion flips role cleanly** — After `pg_ctl promote`, the promoted standby leaves recovery and becomes writable; `pg_is_in_recovery()` becomes false / `in_hot_standby` false. [warm standby](https://www.postgresql.org/docs/18/warm-standby.html), [pg_ctl](https://www.postgresql.org/docs/18/app-pg-ctl.html)
6. **No divergence after settle** — After a fixed workload and wait, both nodes report identical `COUNT(*)`/hash over a test table. This is a good negative property for split-brain or lost-write bugs. [warm standby monitoring](https://www.postgresql.org/docs/18/warm-standby.html)
7. **Standby rejects writes during recovery** — A deliberate write attempt on the standby must error (useful as a safety oracle when testing module-generated configs). [hot standby](https://www.postgresql.org/docs/18/hot-standby.html)
